Why This Comes First
The API modules all build on this same application/package pattern

Parameter, VAPIX, Event, Overlay, VDO, LAROD, FastCGI, and CivetWeb examples all use the same foundation: source files under `app/`, a manifest, a Makefile, an SDK Docker image, and an `.eap` package created by `acap-build`. This intro isolates that foundation so the API chapters can focus on API behavior.

What is the Axis ACAP Intro?
The starting point before learning any specific ACAP SDK API

The intro explains the structure of a native ACAP application: source files, manifest metadata, Docker SDK image, Makefile, `acap-build`, and the generated `.eap` package. The workshop example is intentionally small: it starts, writes to syslog, waits for a stop signal, and exits cleanly.

C Source

`axis_intro_minimal.c` contains the process entry point, syslog messages, signal handling, and the wait loop.

Manifest

`manifest.json` defines package identity, executable name, version, run mode, and AXIS OS compatibility.

Package

`acap-build .` validates the manifest, runs `make`, and creates the installable `.eap` package.

Build Flow
How the SDK container turns source files into an installable `.eap` package
Select the ACAP Native SDK image
ARG ARCH=aarch64
ARG VERSION=12.10.0
ARG UBUNTU_VERSION=24.04
ARG REPO=axisecp
ARG SDK=acap-native-sdk

FROM ${REPO}/${SDK}:${VERSION}-${ARCH}-ubuntu${UBUNTU_VERSION}

# Building the ACAP application
COPY ./app /opt/app/
WORKDIR /opt/app
RUN . /opt/axis/acapsdk/environment-setup* && acap-build .

The SDK image contains the cross compiler, target sysroot, headers, libraries, manifest tooling, and `acap-build`.

Minimal App Anatomy
The smallest workshop app starts, logs, waits for a signal, and stops cleanly
#include <signal.h>
#include <stdbool.h>
#include <stdlib.h>
#include <syslog.h>
#include <unistd.h>

static volatile sig_atomic_t running = 1;

static void signal_handler(int signal_number) {
    (void)signal_number;
    running = 0;
}

int main(void) {
    openlog("axis_intro_minimal", LOG_PID, LOG_USER);
    syslog(LOG_INFO, "Axis intro minimal app started");

    signal(SIGTERM, signal_handler);
    signal(SIGINT, signal_handler);

    while (running)
        sleep(1);

    syslog(LOG_INFO, "Axis intro minimal app stopped");
    closelog();
    return EXIT_SUCCESS;
}

This is the process lifecycle pattern you will see again in larger examples: initialize, register signal handling, keep running, then clean up on stop.
