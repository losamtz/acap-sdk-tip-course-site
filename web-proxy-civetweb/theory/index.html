What is CivetWeb?
Understanding the embedded web server library

CivetWeb is an embeddable C web server library designed for Linux embedded systems. It provides a lightweight, efficient HTTP server that can be integrated directly into ACAP applications, offering a complete web server solution without external dependencies.

Advantages

  • • Embedded directly in application
  • • Low memory footprint
  • • C API for custom extensions
  • • HTTP/HTTPS support
  • • CGI and custom handlers
  • • Cross-platform compatibility

Use Cases

  • • REST API endpoints
  • • Web-based configuration interfaces
  • • File serving and downloads
  • • Real-time data streaming
  • • Custom web applications
  • • Legacy code integration

URL Routing Scheme

"reverseProxy": [
  {
    "apiPath": "api",
    "target": "http://localhost:2001",
    "access": "admin"
  }
]

This configuration routes all requests from /local/web_proxy/api/* to the CivetWeb server running on port 2001.

Reverse Proxy Fundamentals
Understanding the proxy configuration pattern

A reverse proxy configuration allows the Apache server in AXIS OS to forward HTTP requests to your CivetWeb server running inside the ACAP application. This provides a clean, standardized URL structure while maintaining security and authentication.

URL Routing Pattern:

http://<AXIS_DEVICE_IP>/local/<appName>/<apiPath>

Where appName and apiPath are defined in your manifest.json

Core API Functions
Essential CivetWeb functions and request lifecycle for web proxy implementation

1. Initialize + Start Server

// Initialize CivetWeb library
mg_init_library(0);

// Start server with callbacks and options
mg_start(&cb, user_data, opts); // returns mg_context* (the running server)

The server initialization creates the main context that will handle all incoming connections.

2. Connection Handling

// CivetWeb accepts a connection and calls (if set) your server-wide callbacks
cb.begin_request(conn) // you log URI/method and return 0 to continue routing

Each incoming connection triggers the callback chain, allowing you to log and process requests.

3. URI Routing

// CivetWeb matches the request to a registered handler from
mg_set_request_handler(ctx, "/path", handler, cbdata);

// Your handler runs; you read the request and write the response

URL patterns are matched against registered handlers, routing requests to appropriate functions.

4. Finish + Shutdown

// Main loop ends (signal flips running=0)
mg_stop(ctx); // close listeners and worker threads

Graceful shutdown closes all connections and cleans up server resources.
