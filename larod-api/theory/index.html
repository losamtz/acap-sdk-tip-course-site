What is larod?
The local inference service used by ACAP applications

larod is Axis's unified C API and system service for running machine-learning inference and image preprocessing on camera hardware. Applications link with liblarod, then the library talks to the larod service, which schedules work on available backends such as CPU TFLite, DLPU, EdgeTPU, GPU, or preprocessing engines.

Key idea: your app controls the pipeline, but larod owns model execution, backend arbitration, tensor metadata, and optimized access to model buffers.

What it is for
Where larod fits in a camera application

Neural Network Inference

Load backend-specific model binaries, allocate tensors, and run synchronous or asynchronous inference jobs.

Image Preprocessing

Configure crop, scale, and color conversion through a larodMap; preprocessing is loaded and run like any other larod model.

Buffer Efficiency

Use fd-backed tensors, DMA-BUF flags, and tensor tracking to avoid full-frame CPU copies in VDO pipelines.

How it works
Application, service, backend, and camera-frame responsibilities

Service and sessions

Each client gets a session. Private models and session-owned resources are cleaned up when the app disconnects.

Backend units

Backends implement model loading and job execution for the concrete runtime, while larod exposes the same client API.

DMA-BUF, fd properties, and no-copy behavior
The performance-sensitive part of a VDO-to-larod application

READWRITE

Works broadly, but uses read/write paths that normally copy tensor data.

MAP

Allows mmap access. With tensor tracking, larod can cache mappings and avoid repeated setup overhead.

DMABUF

Allows capable backends to receive buffer fds directly. The correct flag matters for cache coherence.

When the app passes externally owned DMA-BUFs, it must set the fd properties honestly. If a backend uses direct DMA-BUF access, userspace synchronization may be the application's responsibility. If a backend maps the DMA-BUF through CPU access, larod can perform the relevant DMA-BUF sync around that mapped access.

int vdo_fd = vdo_buffer_get_fd(buffer);
int64_t offset = vdo_buffer_get_offset(buffer);
size_t capacity = vdo_buffer_get_capacity(buffer);
int tracked_fd = dup(vdo_fd);

larodSetTensorFd(input_tensor, tracked_fd, &error);
larodSetTensorFdOffset(input_tensor, offset, &error);
larodSetTensorFdSize(input_tensor, capacity, &error);
larodSetTensorFdProps(input_tensor, LAROD_FD_PROP_MAP | LAROD_FD_PROP_DMABUF, &error);

// Track recurring VDO buffers once so larod can reuse mappings.
larodTrackTensor(conn, input_tensor, &error);

Usage Example

larodConnection* conn = NULL;
larodError* error = NULL;

if (!larodConnect(&conn, &error)) {
    PANIC("larodConnect: %s", error->msg);
}

const larodDevice* device = larodGetDevice(conn, "a9-dlpu-tflite", 0, &error);
Use larodConnect once during startup before any model, tensor, or job call.
Device names used in the workshop include a9-dlpu-tflite, axis-a8-dlpu-tflite, cpu-tflite, and cpu-proc.
The device handle is passed into larodLoadModel.
Responsibilities and boundaries
What larod owns and what the application still must handle

larod owns

  • • Loaded model handles and backend execution
  • • larod-allocated tensors and job requests
  • • Scheduling across multiple app sessions
  • • Map-once optimization when tensors are tracked

The app owns

  • • VDO stream setup and buffer return
  • • Correct tensor shape, pitch, layout, fd, and fd props
  • • Model-specific postprocessing
  • • Overlay drawing and application control flow
