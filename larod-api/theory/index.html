Understanding Machine Learning Inference on ACAP Devices

Module 0: Introduction for App Developers

Larod is Axis's unified API for efficient on-device inference and image preprocessing through ACAP. The architecture and workflow are designed for flexibility, safety, and performance.

Key Benefits:

  • Unified API: Hides complexity of diverse frameworks (TensorFlow Lite, OpenCL)
  • Multi-app concurrency: Multiple apps can share accelerators safely
  • Asynchronous operations: Non-blocking calls for complex tasks
  • Built-in preprocessing: Resize, color format conversion, cropping
  • Zero-copy optimizations: Efficient memory management with DMABUF

Common Use Cases:

• Object detection
• Image classification
• Anomaly detection
• Face recognition
• Motion analysis
• Quality inspection
Module 1: Architecture & Internals

Larod consists of two principal components: liblarod (client library) and the Larod Service (daemon).

┌─────────────┐ UNIX socket ┌───────────────────┐
│ ACAP App │ ←────────────────────────→ │ Larod Service │
│ (liblarod) │ │ (Backends: CPU...) │
└─────────────┘ └───────────────────┘

High-level Components

  • Service: Runs as systemd service, manages devices
  • Client: Your app using liblarod
  • Session: Per-client context
  • BackEndUnit: Device abstraction layer

Key Features

  • Multi-client: Concurrent app support
  • Async workflow: Queued operations
  • Device abstraction: Hardware independence
  • Resource management: Automatic cleanup
Module 2: Larod Client API

The liblarod client library provides the primary interface for ACAP applications to interact with the Larod service through a well-defined C API.

Core API Functions:

Connection Management:
  • • larodConnect()
  • • larodDisconnect()
Model Operations:
  • • larodLoadModel()
  • • larodLoadModelAsync()
  • • larodUnloadModel()
Job Management:
  • • larodCreateJob()
  • • larodRunJob()
  • • larodRunJobAsync()
Buffer Operations:
  • • larodAllocModelInputs()
  • • larodAllocModelOutputs()
  • • larodSetTensorFd()

Core Concepts:

  • larodConnection: Your session with the service
  • larodDevice: Hardware backend (CPU, DLPU, GPU, etc.)
  • larodModel: Loaded model with private or public access
  • larodTensor: Input/output buffers with FD support
  • larodJobRequest: Combines model + tensors for inference

Understanding model formats, backend capabilities, and buffer management for optimal inference performance.

Buffer FD Properties:

READWRITE

System calls (safe but slow)

MAP

Memory-mapped (fast, preferred)

DMABUF

Zero-copy DMA (most efficient)

ARTPEC-8 DLPU Notes:

  • • First load triggers slow compilation to .nb format
  • • Results cached in /var/lib/larod/nbg-cache
  • • Cache policies: ~30 MiB or 10 models maximum
  • • Preload .nb files to avoid compilation delays
Module 4: Integration & Best Practices

Practical considerations for production deployment, performance optimization, and system integration.

Performance Tips

  • • Prefer mapped buffers (MAP) over READWRITE
  • • Use DMABUF for zero-copy when supported
  • • Enable tensor tracking for buffer reuse
  • • Precompile models for DLPU backends
  • • Use async APIs for non-blocking operations

Integration Patterns

  • • VDO integration for real-time inference
  • • Parameter API for dynamic model switching
  • • Overlay API for result visualization
  • • Event API for trigger-based processing
  • • Proper resource cleanup and error handling

DMABUF Synchronization:

When using DMABUF FDs, applications are responsible for cache management:

  • • Manual flush/invalidation via ioctl(DMA_BUF_IOCTL_SYNC)
  • • Larod handles sync automatically for MAP-only backends
  • • Always query FD properties before buffer allocation
Larod Client CLI Tool

The larod-client CLI utility provides command-line access to the same API your applications use, perfect for testing, benchmarking, and prototyping.

Common Commands:

# List available backends
larod-client -a
# Run inference
larod-client -c cpu-tflite -g model.tflite -i input.bin -o output.bin
# Benchmark performance
larod-client -R 1000 -q -c cpu-tflite -g model.tflite -i input.bin
Key Options:
  • • -d: Debug mode
  • • -c: Specify backend
  • • -R N: Run N iterations
  • • -q: Queue jobs asynchronously
Buffer Types:
  • • -I larod-dma: DMA input
  • • -O larod-dma: DMA output
  • • -X: Track input buffer
  • • -Y: Track output buffer
