What is VDO?
The ACAP video streaming package

VDO consists of two primary components: the vdo service, a video streaming daemon that abstracts the camera video hardware, and the vdo library, a C library used by ACAP applications to communicate with that service.

vdo service

Owns the hardware-facing stream implementation and exposes a unified streaming abstraction across SoCs.

vdo library

Provides GLib/GObject-based C objects and functions for creating streams, fetching buffers, and reading stream metadata.

What it is for
Why VDO is taught before Larod and overlay2

Get camera frames

Request H.264, RGB, or YUV/NV12 frames from a selected channel and resolution.

Share frame memory

Inspect fd-backed buffers and understand the no-copy foundation later used by Larod.

Observe stream lifecycle

Listen for VDO stream events so overlay2 can create overlays for active stream ids.

VDO objects
The objects used by the workshop examples

VdoStream

A client-owned stream. It is configured with a VdoMap and returns VdoBuffer objects.

VdoChannel

A video source, usually one camera input or a composite view. Use it to query supported resolutions.

VdoMap

A variant key/value map used for stream settings, resolution filters, stream info, and event payloads.

VdoBuffer and VdoFrame

The buffer owns frame memory while checked out. The frame exposes timestamp, sequence, and payload size metadata.

VDO is GLib-based. Objects returned as GObject instances must be unreferenced, and failures are reported with GError.

Channel Discovery
Discover which VDO channels exist before selecting stream settings.

Function Signature

GList* vdo_channel_get_all(GError** error);
VdoChannel* vdo_channel_get(guint id, GError** error);
guint vdo_channel_get_id(VdoChannel* self);

Usage Example

GList* channels = vdo_channel_get_all(&error);

for (GList* list = channels; list; list = list->next) {
    VdoChannel* channel = list->data;
    guint id = vdo_channel_get_id(channel);
    syslog(LOG_INFO, "Channel id: %u", id);
}

Stream events for overlay2
The missing VDO concept before the new overlay API

The old overlay path hides most VDO stream management. overlay2 is different: it creates overlays for VDO streams, so the app must know when overlay-capable streams already exist, are created, or are closed.

g_autoptr(VdoStream) event_stream = vdo_stream_get(0, &error);
g_autoptr(VdoMap) stream_filter = vdo_map_new();

vdo_map_set_string(stream_filter, "filter", "overlay");
vdo_stream_attach(event_stream, stream_filter, &error);

int event_fd = vdo_stream_get_event_fd(event_stream, &error);

// In overlay2, this stream id is later used to match and create overlays.
g_autoptr(VdoMap) event = vdo_stream_get_event(event_stream, &error);
unsigned int stream_id = vdo_map_get_uint32(event, "id", 0);
unsigned int event_type = vdo_map_get_uint32(event, "event", 0);

EXISTING

A stream was already active when your app started.

CREATED

A new stream appeared and may need an overlay.

CLOSED

The stream disappeared and associated overlay state should be removed.

Rules to keep
  • • Always read back stream info after creation. VDO may adjust requested settings.
  • • Use a VdoBuffer only between get_buffer and buffer_unref.
  • • Handle transient non-blocking errors such as VDO_ERROR_NO_DATA and VDO_ERROR_NO_EVENT.
  • • Reinitialize streams after interface-down or fatal stream errors.
  • • Teach DMA-BUF after buffer ownership is understood, not before.
