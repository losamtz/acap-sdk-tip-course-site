What is VDO?
The foundation of video capture and production in ACAP applications

VDO is the ACAP Native SDK's video capture API. It's the service your app talks to when it needs frames from the camera pipeline—or when it wants to produce frames back into that pipeline (e.g., test patterns, rendered video). The API is exposed as a set of GLib/GObject types (C API) such as VdoStream, VdoMap, VdoFrame, VdoBuffer, and VdoChannel.

Key Advantage: Direct access to hardware-accelerated video pipeline with zero-copy buffer management for optimal performance.

Core Building Blocks (Mental Model)
Understanding the fundamental components of the VDO API

VdoStream

A session that pulls frames from (or pushes frames to) the camera's video service. You create it with a settings map and then start/stop it.

VdoMap

A key/value dictionary (GObject) for configuring streams and querying info (e.g., format, subformat, width, height, framerate, channel, buffer settings).

VdoBuffer / VdoFrame

The pixel payload and its metadata (sequence number, timestamp, size, frame type, etc.).

VdoChannel

"Which video" a stream connects to (sensor view or composite). Also lets you query supported resolutions for that channel.

VDO Errors

Reported via GError with well-defined codes (e.g., NOT_SUPPORTED, IO, DBUS) for proper error handling and recovery strategies.

Two Usage Modes You Actually Use
Consumer (reader) and Producer (writer) patterns for video processing
R

Consumer (Reader)

Your app pulls frames from the camera pipeline:

  1. 1. Create VdoStream with settings (format=YUV, subformat=NV12, width/height)
  2. 2. Start the stream
  3. 3. Loop: vdo_stream_get_buffer → read data/meta → return via vdo_stream_buffer_unref

This is exactly what your vdo-consumer sample demonstrates.

W

Producer (Writer)

Your app pushes frames into the camera pipeline:

  1. 1. Configure explicit buffers (buffer.strategy=EXPLICIT, buffer.count, write access)
  2. 2. Start the stream
  3. 3. Loop: vdo_stream_buffer_alloc → fill NV12 planes → set frame meta → vdo_stream_buffer_enqueue

Your vdo-producer sample follows this pattern and shows how to compute payload size using vdo_stream_get_info.

Note: The allocation/unref operations are synchronous and can block (design constraint in the API). Keep your producer loop lean with no long CPU work between alloc/enqueue.

Stream Configuration: The Settings Map
How to configure streams using VdoMap key-value pairs

You configure a stream by populating a VdoMap and passing it to vdo_stream_new. Typical keys that appear in the docs and in your code:

Image Format

  • • format (e.g., VDO_FORMAT_YUV)
  • • subformat (e.g., "NV12")

Geometry & Rate

  • • width, height
  • • framerate

Source

  • • channel (which view to read from/produce to)

Buffers (Producer)

  • • buffer.strategy (EXPLICIT)
  • • buffer.count
  • • buffer.access (SDK dependent)

All of these are covered conceptually in the stream header documentation and visible via vdo_stream_get_settings()/get_info().

Stream Information & Negotiation
Querying applied properties and handling configuration negotiation

After start, you can query applied properties using vdo_stream_get_info, for example:

Buffer Information

  • • plane.0.stride, plane.1.stride
  • • How to compute NV12 payload size safely

Stream Properties

  • • width, height, framerate
  • • rotation → actual running configuration

Important: Settings vs. Info - what you request in the settings isn't guaranteed to be what you get; always confirm via vdo_stream_get_info (your utilities sample does this).

Channels and Resolutions (Discovery Path)
Discovering available video channels and supported resolutions

Channel Discovery

Use vdo_channel_get_all() to enumerate available channels (views).

Resolution Discovery

For a given channel, use vdo_channel_get_resolutions() (optionally with filters like aspect_ratio=native) to discover supported dimensions.

That's what your vdo-utilities sample demonstrates - complete channel and resolution discovery workflow.

Error Handling (Why Your GError* err Matters)

Most calls take a GError**. On failure, inspect err->domain / err->code to distinguish conditions and decide whether to retry, downshift quality, or abort.

Design Constraints & Gotchas Worth Mentioning
Important considerations for robust VDO application development

Explicit Buffer API is Synchronous

Alloc/unref can block, so keep your producer loop lean (no long CPU work between alloc/enqueue) and pre-size the buffers according to the negotiated strides.

Channel ↔ Overlay View

If you draw overlays (bbox/axoverlay) on a view, ensure it matches the channel your stream uses, so the right output path shows the boxes. (Overlay API documented separately.)

Settings vs. Info

What you request in the settings isn't guaranteed to be what you get; always confirm via vdo_stream_get_info (your utilities sample does this).

Stream Lifecycle Management

Functions for creating, configuring, starting, and stopping video streams with proper resource management.

vdo_stream_new()
VdoStream* vdo_stream = vdo_stream_new(settings, NULL, &error);
if (!vdo_stream) {
    panic("Failed creating vdo stream: %s", error->message);
}
vdo_stream_start()
if (!vdo_stream_start(image->vdo_stream, &err)) { 
    syslog(LOG_ERR, "Failed to start video stream: %s", err->message);
    return FALSE; 
}
vdo_stream_get_info()
VdoMap* info = vdo_stream_get_info(stream, &err);
syslog(LOG_INFO, "Stream: %ux%u, %u fps",
       vdo_map_get_uint32(info, "width", 0),
       vdo_map_get_uint32(info, "height", 0),
       vdo_map_get_uint32(info, "framerate", 0));
Buffer Management Strategies
Advanced buffer management patterns for optimal performance

Explicit Strategy

Manual buffer allocation and reference management for maximum control.

vdo_map_set_uint32(vdo_map, "buffer.strategy", 
                   VDO_BUFFER_STRATEGY_EXPLICIT);

Queue Management

Thread-safe frame queuing with delivered and processed frame tracking.

g_queue_push_tail(provider->delivered_frames, new_buffer);
old_buffer = g_queue_pop_head(provider->processed_frames);
