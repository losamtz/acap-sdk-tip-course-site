Overlay API Theory

Understanding axoverlay - the ACAP SDK's overlay engine

What is axoverlay?

axoverlay is the ACAP SDK's overlay engine. It lets your app draw graphics (text, lines, shapes, logos, bounding boxes, crosshairs, etc.) on live video streams coming out of the camera. You don't modify the pixels in the main video pipeline; instead you hand the library "overlay content" which is composited on top of each stream.

Think of it as a small rendering service:

  • • You register callbacks that know how to draw
  • • You configure one or more overlay surfaces (size, position, colorspace, anchoring)
  • • The library calls you when a stream needs drawing, and applies your result on that stream

Why use axoverlay?

Real-time UX

Draw detections, regions of interest, UI hints, and watermarks in real-time on live video streams.

Separation of Concerns

Your detection/tracking runs as usual; visualization is decoupled and handled by axoverlay.

Device-native Output

Overlays appear in browsers (VAPIX MJPEG), RTSP/ONVIF viewers, and camera web UIs.

Overlay API Architecture
The axoverlay API provides the rendering system for video stream overlays, while Cairo handles the actual drawing operations within render callbacks.

Architecture Overview

  • axoverlay API: Manages overlay sessions, video stream integration, coordinate systems, and rendering pipeline. Provides the infrastructure for overlays.
  • Cairo Graphics: Drawing library used within render callbacks to create shapes, text, and images. Handles the actual drawing operations.
  • Render Callbacks: Functions called by axoverlay where you use Cairo to draw. The overlay API renders these drawings onto video streams.
  • Stream Integration: axoverlay handles coordinate transformations, multi-stream support, and frame synchronization automatically.

Key Distinction

axoverlay renders your graphics onto video streams and manages the overlay system.Cairo draws the actual shapes, text, and images within your render callbacks. Think of axoverlay as the canvas system and Cairo as your paintbrush.

Cairo Image Backend

A software 2D drawing API (AXOVERLAY_CAIRO_IMAGE_BACKEND). Simple and predictable, great for lines, rectangles, and text.

// Set Cairo backend in settings
settings.backend = AXOVERLAY_CAIRO_IMAGE_BACKEND;
settings.render_callback = my_render_callback;
Step-by-step Implementation Flow
Detailed implementation guide from initialization to cleanup

1) Initialization & Backend Choice

Create settings struct, set render callback, select Cairo backend, and call axoverlay_init.

// Create and initialize settings
struct axoverlay_settings settings;
axoverlay_init_axoverlay_settings(&settings);

// Set the render callback (mandatory)
settings.render_callback = my_render_callback;

// Select Cairo as backend
settings.backend = AXOVERLAY_CAIRO_IMAGE_BACKEND;

// Initialize axoverlay once at startup
GError* error = NULL;
if (!axoverlay_init(&settings, &error)) {
    g_error("Failed to initialize axoverlay: %s", error->message);
    g_error_free(error);
}

// What this achieves: axoverlay is ready to create surfaces

2) Configure an Overlay Surface

Create overlay data, decide placement, size, and colorspace.

// Create and initialize overlay data
struct axoverlay_overlay_data data;
axoverlay_init_overlay_data(&data);

// Decide placement (normalized coordinates recommended)
data.postype = AXOVERLAY_CUSTOM_NORMALIZED;
data.anchor_point = AXOVERLAY_ANCHOR_CENTER;  // Anchor reference
data.x = 0.0;   // Center horizontally (-1..+1)
data.y = -0.8;  // Near bottom (-1..+1)

// Decide size (surface pixel dimensions)
data.width = 300;
data.height = 200;

// Decide colorspace
data.colorspace = AXOVERLAY_COLORSPACE_ARGB32;

// Optional: auto-scale to preserve aspect ratio
data.scale_to_stream = true;
Scaling and Aspect-ratio: Three Strategies
Choose the right approach for your use case
Fixed Resolution Only

Use select callback to accept only streams with your intended resolution.

Result: Overlay looks exact or not shown at all
Manual Coordinate Adaptation

Keep surface large; adjust drawing coordinates based on axoverlay_stream_data in render callback.

Result: Full control but requires more code
Auto-scale (Simplest)

Set data.scale_to_stream = true. Surface scales while preserving aspect ratio.

Result: Content stays aligned with physical objects across resolutions
The Callback Family (When and Why)
Understanding the different callback types and their purposes

render_callback (required)

Invoked to draw on a stream's overlay surface (new stream or redraw). This is where you call Cairo to render shapes/text.

When: New stream appears or axoverlay_redraw() called

select_callback (optional)

Invoked when a new stream is created. Return true/false to enable/disable overlay for that stream.

Use for: Filter by rotation, resolution, codec; disable on low-bandwidth streams

adjustment_callback (optional)

Runs before every render; allows changing overlay placement dynamically. Use sparingly for performance.

Prefer: Drawing inside larger surface or using scale_to_stream
Best Practices & Troubleshooting
Essential guidelines for successful overlay implementation

Performance & Efficiency

  • • Keep render fast: Cairo is CPU-based; minimize allocations
  • • Cache resources: Colors, paths, and fonts if possible
  • • Prefer outlines: Simple shapes over filled translucent areas
  • • Throttle redraws: Don't redraw every frame

Design & Architecture

  • • Normalize placement: Use normalized + scale_to_stream
  • • Separate concerns: Treat render callback as a "view"
  • • Handle rotation: Use stream info to adapt orientation
  • • Choose Cairo: Easiest for workshops and simple graphics

Resource Management

  • • Handle errors: Every axoverlay function can fail; check GError*
  • • Destroy what you create: Track overlay IDs and cleanup properly
  • • Clear errors: Always g_error_free() to avoid leaks

Summary

axoverlay provides a clean, device-native way to overlay Cairo-drawn graphics on live streams.

You configure where a surface lives (normalized placement + anchor) and how big it is (pixels), then draw in Cairo pixel coordinates inside that surface.

Use scale_to_stream to keep drawings visually consistent across resolutions, and the render/select/adjustment callbacks to control when and how overlays appear.

Keep the render fast, manage resources carefully, and let axoverlay handle the compositing so your app can focus on analytics and logic.
