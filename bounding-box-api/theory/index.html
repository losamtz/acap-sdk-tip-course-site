Bounding Box API Theory

Understanding the lightweight overlay library for drawing simple shapes on video output

What is the BBox API?

A lightweight overlay library for drawing simple shapes—mainly bounding boxes—on a device's video output. It's designed to be the fast, easy path when you only need boxes and basic lines, while the fuller Axoverlay API covers more advanced overlay use cases.

Core Capabilities

  • • Lightweight overlay rendering
  • • Simple shapes (boxes, lines, paths)
  • • Frame-aligned or scene-aligned coordinates
  • • Multi-channel support

Performance Benefits

  • • Low overhead rendering
  • • Direct mapping to camera views
  • • Deferred drawing with commit
  • • Timestamp synchronization

What it's for

Detection Visualization

Visualizing detections and regions over live video, such as model outputs and regions of interest (ROIs).

  • • Model output visualization
  • • ROI highlighting
  • • Detection confidence display
  • • Real-time overlay updates
Low Overhead Rendering

Efficient rendering that maps directly to the camera's views and channels with minimal performance impact.

  • • Direct camera mapping
  • • Multi-channel support
  • • Optimized for real-time
  • • Minimal CPU overhead
Coordinate Flexibility

Frame-aligned or scene-aligned coordinates, letting you choose whether boxes track frame orientation or physical scene.

  • • Scene-normalized coordinates
  • • Frame-normalized coordinates
  • • Rotation independence
  • • Semantic positioning

How it works (Conceptual Flow)

1

Create Session

Create a bbox_t handle bound to one or more views/channels using bbox_new() or bbox_view_new().

2

Pick Coordinate System

Choose scene-normalized (default) or frame-normalized coordinates for your overlay positioning.

3

Set Style

Configure color, outline vs corners style, and line thickness for subsequent geometry.

4

Queue Geometry

Add rectangles, lines, quads, or paths using normalized coordinates. Nothing is drawn yet.

5

Commit

Call bbox_commit() to render all queued geometry with optional timestamp synchronization.

6

Update Loop

Repeat clear → enqueue shapes → commit as your detections change over time.

7

Tear Down

Free the session with bbox_destroy() when finished to release all resources.

Session Types
  • • Global: Multi-channel session
  • • View: Single view session
  • • Lifetime: Create → Use → Destroy
Coordinate Systems
  • • Scene-normalized: [0..1] × [0..1] scene space
  • • Frame-normalized: [0..1] × [0..1] frame space
  • • Clipping: Auto-clipped to visible range
Rendering Model
  • • Deferred: Queue first, render on commit
  • • Synchronized: Timestamp alignment
  • • Efficient: Batch rendering

Performance Benefits

  • • Low overhead rendering pipeline
  • • Direct mapping to camera views/channels
  • • Minimal CPU usage for simple shapes
  • • Deferred drawing reduces render calls
  • • Timestamp synchronization with video
  • • Optimized for real-time applications

When to Choose BBox vs Axoverlay

Choose BBox When
  • • You need simple, performant boxes and lines
  • • Minimal setup and configuration required
  • • Low overhead rendering is critical
  • • Basic geometric shapes are sufficient
  • • Real-time performance is essential
Choose Axoverlay When
  • • You need richer overlay features
  • • Complex graphics and text rendering
  • • Advanced styling and effects
  • • Custom drawing operations
  • • Beyond basic geometric shapes
