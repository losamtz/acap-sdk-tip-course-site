Parameter API Theory

Master ACAP's persistent configuration system for robust, user-friendly applications

What is AXParameter?
The foundation of persistent configuration in ACAP applications

AXParameter is the ACAP Native SDK's C library for managing your application's persistent settings on an Axis device. It lets an app define parameters (with defaults), read/update them at runtime, and react immediately to changes—even when those changes come from the device's web UI or VAPIX.

Crucially: Values survive app restarts, reboots, and firmware upgrades.

What it's for
Three core use cases that make AXParameter essential

Persisting Configuration

Store app-specific settings that must not be lost across reboots or OS upgrades.

Unified Control Surface

Users can change parameters via your app's settings page or via VAPIX; your app receives callbacks and can apply changes on the fly.

Bridging Manifest & Runtime

Start with parameters declared in manifest.json, and add/remove parameters dynamically from C as your app evolves.

Continuity across ACAP generations: The API was a staple in ACAP 3 and (by developer demand) returned to ACAP 4, enabling straightforward migration paths.

How it works (conceptually)
The step-by-step workflow of parameter management
1

Create a parameter handle

Think of it as a session to the parameter store for your app.

2

Declare parameters

  • • Pre-declare in manifest.json (names, defaults, types)
  • • Or add at runtime (with default value and metadata)
3

Get/Set values

Read current values or update them. Writes persist in the camera's configuration store.

4

Register callbacks

Subscribe to parameter-change events; your function is invoked when a value changes (e.g., user edits a setting in the web UI), so you can validate, reconfigure pipelines, or restart services seamlessly.

5

List & manage

Enumerate existing parameters, remove obsolete ones, or introspect available keys to drive UIs.

6

Tear down

Free resources cleanly when shutting down.

Key building blocks in the library
Understanding the core components of the Parameter API

Parameter context (AXParameter)

The central object you create for your app; it scopes all get/set/add/remove operations.

Application parameters

Namespaced key–value pairs belonging to your app. They can be seeded from manifest.json and extended at runtime. Stored persistently.

System parameters

Selected device parameters your app may query to adapt behavior (e.g., environment, model specifics). Read-only or limited write.

Callbacks (observer pattern)

Functions you register that run when a parameter is updated, regardless of the origin (your app, settings page, or VAPIX). Enables reactive configuration with no polling.

Error handling (GLib)

All operations report status via GError and return booleans, matching the broader GObject/GLib conventions used across ACAP Native SDK.

Typical lifecycle & responsibilities
From design to deployment: managing parameters throughout your app's lifetime

Design time

Decide which knobs your app needs. Define stable names and defaults in manifest.json; decide types and validation rules.

Startup

Create the parameter handle; add missing parameters with defaults if you're evolving the schema over time; register callbacks for the ones you want to observe; read initial values to configure subsystems.

Runtime

When a parameter changes, your callback runs—validate, clamp, reconfigure (e.g., restart a pipeline with a new resolution), and optionally write back a corrected value.

Shutdown/upgrade

Just free the handle. Values remain in persistent storage so your app comes back with the same settings after restart or OS upgrade.

Why it matters in ACAP apps
The strategic advantages of using AXParameter in your applications

Resilience

Users don't lose settings during updates; support teams can rely on persistent state.

Consistency

One authoritative place for configuration—aligned with Axis UI and VAPIX.

Maintainability

Clear API surface—create, list, get, set, remove, register/unregister callbacks—keeps config logic predictable and testable.

Defining Parameters in manifest.json
Pre-define parameters with defaults, types, and validation rules in your application manifest

Define parameters in your manifest.json file to pre-configure application settings with defaults and validation.

{
    "schemaVersion": "1.7.4",
    "resources": {
        "linux": {
            "user": {
                "groups": [
                    "admin"
                ]
            }
        }
    },
    "acapPackageConf": {
        "setup": {
            "friendlyName": "parameter manifest",
            "appName": "parameter_manifest",
            "vendor": "Andrés",
            "embeddedSdkVersion": "3.0",
            "vendorUrl": "https://www.axis.com",
            "runMode": "never",
            "version": "1.0.0"
        },
        "configuration": {
            "paramConfig": [
                {
                    "name": "ParameterManifest",
                    "default": "no",
                    "type": "string"
                }
            ]
        }
    }
}
Critical Security & Threading Notice

Application parameters are not private

They can be accessed by:

  • • Operator users: Read access via VAPIX
  • • Admin users: Read and write access via VAPIX
  • • Same group applications: Read and write access

Deadlock Warning

Callback functions should avoid blocking calls. Never call any `ax_parameter_*` function from within a parameter callback, as this will very likely lead to a deadlock. A better way to handle updates of other parameters from a callback is to use a global data object that gets updated, and process it in your main application loop.

Parameter Control Words
When creating a parameter, you can prepend a control word to its type to change how the system handles it.
Hidden
type = "hidden:string"

Hides the parameter from the list in the camera's ACAP settings interface.

NoSync
type = "nosync:int"

Changes to this parameter are not saved to persistent storage and will be lost on restart.

ReadOnly
type = "readonly:bool:no,yes"

The parameter cannot be modified via the API or the web UI, where it will appear grayed out.

Core API Functions
A reference for the essential functions in the Parameter API, based on the official `ax_parameter.h` documentation.

Lifecycle Management

Create and destroy parameter handles

ax_parameter_new
Create a new parameter handle for the specified application.

Function Signature

AXParameter* ax_parameter_new(const gchar* app_name, GError** error);

Usage Example

AXParameter* handle = ax_parameter_new("my_app", &error);
if (handle == NULL) {
    g_error("Failed to create parameter handle: %s", error->message);
}
ax_parameter_free
Free the parameter handle and all associated resources.

Function Signature

void ax_parameter_free(AXParameter* param);

Usage Example

ax_parameter_free(handle);
