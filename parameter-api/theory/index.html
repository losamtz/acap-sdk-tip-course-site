Parameter API Theory

AXParameter for persistent ACAP application settings, VAPIX access, callbacks, and custom setting pages.

What is AXParameter?
The ACAP C library for application parameters

AXParameter lets an ACAP application read and modify parameters declared in manifest.json, add and remove parameters from C code, react immediately to changes through callbacks, and read permitted system parameters.

Application parameters are persistent. They survive application restart, application upgrade, device restart, and firmware upgrade.

What it is for
The practical roles parameters play in the workshop examples

Manifest defaults

Declare stable settings in manifest.json before app start.

Runtime parameters

Add, set, list, and remove parameters dynamically in C.

Callbacks

React to changes made from the UI, VAPIX, or code.

Custom UI

Use param.cgi from an app setting page to update values.

Application settings page and VAPIX
How users and external systems interact with app parameters

Axis device web page

  • • Go to the Apps tab.
  • • Open the application options.
  • • Click Settings to open the parameter dialog.
  • • Reload the web page to display values changed from C code or VAPIX.

VAPIX param.cgi

# List app parameters
curl --anyauth -u root:pass \
  "http://CAMERA_IP/axis-cgi/param.cgi?action=list&group=root.parameter_manifest"

# Update one app parameter
curl --anyauth -u root:pass \
  "http://CAMERA_IP/axis-cgi/param.cgi?action=update&root.parameter_manifest.ParameterManifest=yes"

Visibility and callback warnings

Application parameters are not private

  • • Operator users can read them through VAPIX.
  • • Admin users can read and modify them through VAPIX.
  • • Another application can read and modify them if the application users belong to the same group.
  • • Values can appear in the device audit log unless the parameter type is password or writeonly.

Callback safety

Keep callbacks short. The custom-interface workshop example defers writes with g_timeout_add_seconds so the callback can return before another parameter is written.

Parameter properties

Persistence

Values are preserved across app restarts, app upgrades, device restarts, and firmware upgrades.

Control surfaces

Values can be displayed and set in the application settings page, and read or modified with VAPIX param.cgi.

Core API Functions
Workflow-oriented signatures and examples based on the parameter workshop.

1. Lifecycle

Open the application parameter scope and clean it up on shutdown.

ax_parameter_new
Create an AXParameter handle for the application scope.

Function Signature

AXParameter* ax_parameter_new(const gchar* app_name, GError** error);

Usage Example

GError* error = NULL;
AXParameter* handle = ax_parameter_new(APP_NAME, &error);
if (handle == NULL) {
    panic("%s", error->message);
}
ax_parameter_free
Release the parameter handle. Stored values remain persistent.

Function Signature

void ax_parameter_free(AXParameter* param);

Usage Example

g_main_loop_unref(loop);
ax_parameter_free(handle);
